# 비개발자를 위한 바이브 코딩 튜토리얼

> 코드를 한 줄도 직접 쓰지 않고, **AI(Claude Code)와 한국어로 대화**하며 이 프로젝트(한빛앤 활성화 플랫폼 기획·목업·개발 계획·아키텍처)를 만든 전 과정을 그대로 따라 할 수 있게 정리했습니다.

---

## 1. 바이브 코딩(Vibe Coding)이란?

**"무엇을 원하는지 말로 설명하면, AI가 코드를 만들고 검증까지 해주는 방식"** 입니다.

- 당신의 역할: **무엇을(What)·왜(Why)** 를 설명하고, 결과를 확인하고, 방향을 정한다.
- AI의 역할: **어떻게(How)** — 실제 파일 작성, 코드, 검증, 문서화, Git 작업을 수행한다.

> 운전에 비유하면 — 당신은 "목적지"를 말하는 사람이고, AI는 운전대를 잡습니다. 길은 AI가 찾지만, **어디로 갈지와 도착했는지 확인하는 건 당신**입니다.

## 2. 이 프로젝트에서 실제로 만든 것

대화만으로 아래를 전부 만들었습니다.

| 순서 | 만든 것 | 한 일 |
|------|---------|-------|
| 1 | **활성화 대시보드 목업** | PRD(기획서)를 보고 6화면짜리 인터랙티브 화면을 자동 제작 |
| 2 | **목업 개선** | "더 나은 방법" 자문 → 필터가 실제로 작동하는 데모로 업그레이드 |
| 3 | **개발 계획서 + GitHub 이슈 15개** | 개발자들이 바로 작업할 수 있도록 작업을 쪼개 등록 |
| 4 | **아키텍처 문서 + 다이어그램** | 시스템 구조를 그림(Mermaid)과 함께 문서화 |
| 5 | **Git 커밋·푸시** | 결과물을 안전하게 저장하고 GitHub에 업로드 |

## 3. 준비물

1. **Claude Code** 가 설치된 컴퓨터 (터미널 또는 데스크톱 앱)
2. 작업할 **폴더 하나** (예: 기획서가 들어있는 폴더)
3. 그게 전부입니다. 프로그래밍 지식은 필요 없습니다.

## 4. 따라 하기 — 실제로 사용한 프롬프트

각 단계에서 **그대로 입력한 한국어 프롬프트**를 보여드립니다. 복사해서 응용하세요.

### 단계 1. 기획서를 화면으로 만들기

```
완성된 PRD에 기반해서 목업을 랄프 루프를 사용해서 만들어줘
```

- **PRD** = 제품 기획서. **목업** = 실제로 보이는 화면 견본. **랄프 루프** = "끝까지 알아서 완성하고 스스로 검증하는" 자동 작업 모드.
- AI가 기획서를 읽고 → 화면 6개를 만들고 → 브라우저에서 에러가 없는지 직접 검증한 뒤 완성했습니다.

💡 **포인트**: "랄프 루프 써줘"처럼 **작업 방식**을 지정하면, AI가 중간에 멈추지 않고 검증까지 끝냅니다.

### 단계 2. 더 좋은 방법이 있는지 물어보기

```
너가 봤을 때, 더 나은 방법이 있다면 제안해줘
```

- AI가 **무조건 시키는 대로만 하지 않고**, "이 제품은 측정이 핵심인데 목업의 숫자는 가짜다, 진짜 가치는 다른 곳에 있다"는 솔직한 조언을 했습니다.
- 비개발자에게 가장 강력한 프롬프트 중 하나입니다. **"이게 최선이야?"** 라고 묻는 습관을 들이세요.

### 단계 3. 목적을 정하고 개선하기

AI가 목적을 되물었고("이해관계자 설득용? 실제 개발용?"), 답을 고르자 거기에 맞춰 **필터가 실제로 작동하는 데모**로 업그레이드했습니다.

💡 **포인트**: AI가 선택지를 물으면, **목적을 분명히** 답해주세요. 결과물의 방향이 완전히 달라집니다.

### 단계 4. 개발자가 일할 수 있게 정리하기

```
이 프로젝트를 개발자들이 개발할 수 있도록 계획문서를 작성하고
계획 문서의 항목들을 깃헙 이슈에 등록해줘. 깃헙 이슈에는 작업 배경,
작업 내용, 인수 조건, 다른 작업과의 의존성이 포함 되어야 해.
```

- AI가 큰 목표를 **15개의 작은 작업**으로 쪼개고, 각각을 GitHub 이슈로 등록했습니다.
- **인수 조건**(= "이 작업이 끝났다고 인정하는 기준")처럼 **원하는 형식을 구체적으로 지정**하면 그대로 만들어 줍니다.

💡 **포인트**: 원하는 결과의 **구성 요소를 나열**하면(배경/내용/조건/의존성), 품질이 크게 올라갑니다.

### 단계 5. 문서와 그림 만들기

```
아키텍처 문서 생성해줘. 다이어그램은 mermaid로 그려줘. 리드미 업데이트 해줘.
```

- 시스템 구조 문서 + **그림(다이어그램)** 까지 자동 생성. "mermaid로 그려줘"처럼 **도구를 지정**할 수도 있고, 그냥 "그림으로 그려줘"라고 해도 됩니다.

### 단계 6. 결과물 저장하고 업로드하기

```
작업 내용 모두 커밋하고 푸시해
```

- **커밋** = 저장, **푸시** = GitHub에 업로드. (자세한 건 → [비개발자를 위한 Git 매뉴얼](git-manual-for-non-developers.md))

## 5. 비개발자를 위한 5가지 핵심 원칙

1. **작게 요청하라** — "전부 다 만들어줘"보다 한 단계씩. 결과를 보고 다음을 정합니다.
2. **목적을 말하라** — "왜 필요한지"를 알려주면 AI가 더 똑똑하게 판단합니다.
3. **검증을 요청하라** — "정말 되는지 확인해줘", "테스트해줘"라고 하면 증거를 가져옵니다.
4. **모르면 물어라** — "이거 무슨 뜻이야?", "쉽게 설명해줘"는 언제든 환영입니다.
5. **되짚어라** — "더 나은 방법 없어?", "이게 최선이야?"로 품질을 끌어올립니다.

## 6. 자주 쓰는 프롬프트 모음 (복사용)

| 하고 싶은 것 | 이렇게 말하세요 |
|--------------|-----------------|
| 무언가 만들기 | `~~를 만들어줘. 다 되면 정말 작동하는지 확인해줘` |
| 끝까지 자동 완성 | `랄프 루프로 ~~를 완성해줘` |
| 더 좋은 방법 찾기 | `더 나은 방법이 있다면 제안해줘` |
| 결과 확인 | `방금 만든 거 브라우저에서 열어서 보여줘` / `테스트 돌려줘` |
| 쉽게 설명 듣기 | `방금 한 작업을 비개발자도 이해하게 설명해줘` |
| 문서로 정리 | `지금까지 한 걸 문서로 정리해줘` |
| 저장·업로드 | `변경 내용 커밋하고 푸시해줘` |

## 7. 막혔을 때

- **결과가 이상해요** → `방금 결과가 예상과 달라. 뭐가 문제인지 확인하고 고쳐줘`
- **용어를 모르겠어요** → `방금 말한 "○○"가 뭔지 비유로 설명해줘`
- **너무 많이 바뀌었어요** → `방금 변경을 되돌리고, 더 작게 다시 해줘`
- **이게 맞나 불안해요** → `이대로 진행해도 괜찮은지, 위험한 점은 없는지 알려줘`
